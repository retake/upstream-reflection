# プロジェクト計画書：上流反映レビュー スキル

> 本ドキュメント自体が「上流資料」です。各フェーズで判明したことは、この計画書へ反映していきます（このスキルの思想のドッグフーディング）。

- バージョン: 0.1（ドラフト）
- 最終更新: 2026-05-25
- ステータス: Phase 0（計画確定）

---

## 1. 目的・背景

プログラム管理の過程で生まれる**下流の成果物（アウトプット）**を、その**上流資料**と突き合わせ、
下流の内容から「上流に反映すべき候補」を抽出してレビュー表として提示するスキルを作る。

下流の設計・検討を進めると、上流資料の前提・スコープ・要件が陳腐化・矛盾・不足することが常に起きる。
人手だと見落とし・後戻りの温床になるため、これを半自動で洗い出し、トレーサビリティを保つことが狙い。

将来的には複数のレベルペアに拡張する:
- 要件定義 → プロジェクト計画書
- 基本設計 → 要件定義書 ★初期スコープ
- 詳細設計 → 基本設計 / 要件定義

## 2. スコープ

### 初期スコープ（今回作り込む範囲）
- **対象ペア: 基本設計（下流）→ 要件定義書（上流）の1ペアのみ**
- **出力: レビュー表**（候補一覧。列構成は §6）
- **入力: 混在形式**（Office / Markdown / テキスト / Wiki エクスポート）に対応
- **検証: 合成サンプル**で実施（実資料はこのPCでは扱わない）
- **配布前提**: 当PCは「作るだけ」。実運用は業務PCの業務用 Claude Code

### 非スコープ（今回やらない）
- 他レベルペアの作り込み（構造は拡張可能にするが中身は後続）
- 上流資料の自動上書き編集（出力はレビュー表まで。反映の最終判断は人間）
- 機密データの取り扱い・実資料での自動学習

## 3. 成功基準（KPI）

| 観点 | 指標 | 目標（初期） |
|---|---|---|
| 精度 | 提示候補の採用率（採用＋修正採用）/ 全候補 | ≧ 60% で開始、運用で向上 |
| 網羅性 | 重要な反映漏れ（後工程で発覚した抜け）件数 | 合成サンプルで 0 を目指す |
| 効率 | 手動レビュー比の時間短縮 | 体感で半減 |
| 雑音 | 「却下（実装詳細で反映不要）」の割合 | 運用で逓減 |
| 配布性 | 業務PCへ設置後、追加設定なしで動くか | 設置手順のみで動作 |

## 4. 配布・ポータビリティ方針

- **自己完結**: スキルディレクトリ一式（後述）だけで動作。個人の絶対パス・氏名・メール等を埋め込まない。
- **ツール非依存＋グレースフル**: 混在形式の読取は Claude Code 標準の Read を第一手段とし、Office等は
  「変換ツールがあれば使う／無ければユーザにエクスポート（PDF/Markdown）を案内」する設計（`references/ingestion.md`）。
- **学習の共有モデル（ハイブリッド）**:
  1. 配布物に**ベースライン知識**（判断基準・事例）を同梱しバージョン管理。
  2. 各PCは**ローカルのフィードバックログ**を溜める（配布物の外）。
  3. 定期的に**蒸留**し、メンテナがレビューのうえ次版の同梱知識へ反映 → 再配布。
- **検証可能性**: 合成サンプルで「期待される候補」を定義し、設置後に受け入れ確認できる。

## 5. パッケージ構成（配布物）

```
upstream-reflection/              ← この1フォルダを業務PCの ~/.claude/skills/ に置く
  SKILL.md                        ← トリガ記述＋ワークフロー手順
  references/
    taxonomy.md                   ← 種別・優先度・確信度の定義
    criteria.md                   ← 反映すべき/すべきでないの判断基準（成長する知識）★
    checklist.md                  ← 見落とし防止の観点チェックリスト
    examples.md                   ← 良例/反例の事例集（few-shot, 成長する知識）★
    ingestion.md                  ← 混在形式の読取・変換ガイド
  templates/
    review-table.md               ← レビュー表テンプレ
  feedback/
    README.md                     ← フィードバックの記録方法と蒸留手順
    feedback-log.template.md      ← ローカルで使うログ雛形
  CHANGELOG.md                    ← 同梱知識のバージョン履歴

```

## 6. レビュー表の列構成（出力仕様）

| 列 | 内容 |
|---|---|
| ID | 連番（追跡用） |
| 種別 | 追加 / 変更 / 矛盾(要修正) / 明確化 / 削除・縮小 / 影響波及 |
| 対象箇所（上流） | 要件定義書の章・項目（無い場合は「記載なし」） |
| 現状 | 要件定義書の現記述（または「記載なし」） |
| 反映案 | 提案する追記・修正文 |
| 下流根拠 | 基本設計のどこに何が書かれているか（トレーサビリティ） |
| 優先度 | 高 / 中 / 低（影響度 × 緊急度） |
| 確信度 | 高 / 中 / 低（本当に上流反映すべきかの自信） |
| 影響・波及 | 関連要件・他成果物・コスト/スケジュール/リスクへの波及 |
| 判断理由 | なぜ反映要（または不要）と判断したか |
| 採否 | 採用 / 修正採用 / 却下（ユーザが記入。学習ループの入力） |
| FB理由 | 採否の理由（ユーザが記入。蒸留の材料） |

## 7. 判定タクソノミー（骨子。詳細は taxonomy.md / criteria.md へ）

「基本設計に書かれているが要件定義に無い/食い違う」ことの**上流反映要否**を判断する。

**上流（要件定義）へ反映すべき例**
- 設計で判明した新規の機能要件・非機能要件（要件定義に未記載）
- 非機能要件の具体化（性能・可用性・セキュリティの数値が設計で確定）
- 外部連携・データ要件の追加、業務フロー・例外ケースの新発見
- 要件の曖昧さを設計で確定させた前提・制約（明確化）
- 要件定義との矛盾（設計上実現不可 → 要件側の変更が必要）
- スコープの増減

**上流へ反映「不要」な例（設計内に留める＝却下）**
- 純粋な実現手段の詳細（画面レイアウト具体、テーブル設計、内部処理ロジック）
- 要件に影響しない技術選定

→ この境界線の判断こそが精度の核であり、フィードバックで継続的に磨く対象。

## 8. フェーズ計画

各フェーズに「成果物」と「完了条件（exit）」を定義。完了条件を満たしたら次へ。

### Phase 0 — 計画確定 ✅(本書)
- 成果物: 本 PROJECT_PLAN.md
- exit: スコープ・出力仕様・配布方針・学習方式の合意

### Phase 1 — ドメイン設計（知識ベースの骨格）
- 成果物: `references/taxonomy.md`, `references/criteria.md`(初版), `references/checklist.md`, `templates/review-table.md`
- exit: 「反映すべき/すべきでない」の判断基準とレビュー表テンプレが定義済み

### Phase 2 — スキル本体MVP
- 成果物: `SKILL.md`（frontmatter＋ワークフロー：資料特定→トレーサビリティ整合→抽出→分類→表生成）, `references/ingestion.md`
- exit: 入力2資料を与えるとレビュー表が出る最小版が動く

### Phase 3 — 合成サンプルで検証・調整
- 成果物: `samples/`（要件定義書＋基本設計のミニペア。意図的に「反映すべき差分」と「反映不要な詳細」を仕込む）＋ 期待候補リスト、調整済み `criteria.md`, `examples.md`(初版)
- exit: 合成サンプルで期待候補を概ね検出、雑音が許容範囲

### Phase 4 — 精度向上の仕組み（学習ループ）
- 成果物: `feedback/README.md`, `feedback/feedback-log.template.md`, `CHANGELOG.md`, 蒸留手順
- exit: 「採否記録 → 蒸留 → criteria/examples 更新提案 → 次版反映」の運用が文書化され、合成例で一周回せる

### Phase 5 — 配布準備（ポータビリティ）
- 成果物: 導入手順README（業務PCへの設置、~/.claude/skills/ 配置、任意ツール）、個人情報/パス依存の除去確認、受け入れチェック
- exit: 配布フォルダをコピーするだけで業務PCで動作確認できる状態

## 9. リスクと対応

| リスク | 対応 |
|---|---|
| 業務PCで実資料を当方が確認できない | 合成サンプルで受け入れ基準を定義。設置者が自分の資料で確認できる手順を同梱 |
| 混在形式（特にOffice）の読取が環境依存 | Read優先＋変換のグレースフル設計。最終手段はPDF/Markdownエクスポート案内 |
| 「実装詳細」と「要件反映事項」の誤判定（雑音/漏れ） | criteria＋事例の継続学習。確信度を必ず付与し人間が最終判断 |
| 配布後に学習が分散して品質がばらつく | ハイブリッド学習：ローカル蓄積→メンテナ蒸留→同梱知識を版管理し再配布 |
| 機密情報の混入・流出 | スキルは資料を外部送信しない。ローカルログにも機密本文を貼らない運用ルールを明記 |
| 大規模資料でのコンテキスト超過 | 章単位の分割レビュー手順を SKILL.md に用意 |
