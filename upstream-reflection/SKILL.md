# 上流反映レビュー（upstream-reflection）

下流の成果物を読み、上流資料へ**反映すべき候補**をレビュー表として提示するスキル。
判断は人間が行えるよう、各候補に**根拠・優先度・確信度・判断理由**を必ず付ける。

## 対応ペア
- **基本設計（下流）→ 要件定義書（上流）** … 現バージョンの対象。
- 他ペア（要件定義→計画書、詳細設計→基本設計 等）は `references/criteria.md` を増設して拡張する。

## 原則（必ず守る）
1. **捏造しない**: 下流資料に根拠がない候補を作らない。各候補に出典（章・項）を示す。
2. **What と How を分ける**: 要件（What/制約/前提）は上流候補に、実現手段（How）は設計内に留める（`references/criteria.md` §7）。
3. **確信度を必ず付ける**: 不確かなものは確信度=中で出し、判断は人間に委ねる。
4. **自動で上流を書き換えない**: 出力はレビュー表まで。反映の最終決定は人間。
5. **機密を外部に出さない**: 資料を外部サービスに送らない。変換はローカルで行う。ログに機密本文を貼らない。

## ワークフロー

### Step 1. 入力の確認
- 下流資料（基本設計）と上流資料（要件定義書）の場所/内容を特定する。どちらがどちらか不明なら確認する。
- レベルペアが「基本設計→要件定義」かを確認（違うペアなら、対応する criteria が未整備である旨を伝える）。

### Step 2. 読み取り（混在形式対応）
- `references/ingestion.md` に従って両資料をテキスト化する。
- Markdown/テキスト/PDF は Read で直接。Office/Wiki は変換（無ければユーザにエクスポートを案内）。
- 大規模な場合は**章単位に分割**して進める。

### Step 3. 構造把握・トレーサビリティ
- 上流（要件定義）の章立て・要件項目を一覧化する。
- 下流（基本設計）の章立てを把握し、上流項目との**対応関係**をざっくりマッピングする。
- これにより「上流に対応箇所が無い＝追加候補」を見つけやすくする。

### Step 4. 抽出
- `references/checklist.md` のカテゴリ（機能/非機能/セキュリティ/データ/連携/業務フロー/前提・制約/用語）を順に走査。
- 各記述に `references/criteria.md` のシグナル **S1〜S10** を当てはめ、該当を候補として拾う。
- まず**レベル判定**（要件レベルか実装詳細か）を行い、実装詳細は §7 の除外ルールで落とす。

### Step 5. 分類・評価
- 各候補に **種別 / 優先度 / 確信度 / 判断理由**を付与（`references/taxonomy.md`）。
- 影響・波及（コスト・スケジュール・リスク・他要件）を記す。
- グレーゾーンは確信度=中で、判断理由に両論を書く。

### Step 6. 出力
- `templates/review-table.md` の形式でレビュー表を生成する。
- **確信度=高 → 中**の順に並べ、低は末尾か折りたたみ。
- 末尾に簡単なサマリ（高優先の要対応、反映漏れリスク、判断を仰ぎたい論点）を付ける。

### Step 7. フィードバック誘導
- ユーザに各行の「採否（採用/修正採用/却下）」と「FB理由」の記入を促す。
- 記録方法は `feedback/README.md` を案内。これが精度向上ループの入力になる。

## 確信度の付け方（再現性のため）
- 「要件レベルである根拠」と「上流に欠落/矛盾している根拠」の**両方が明示できる** → 高
- どちらか一方が弱い/文脈依存 → 中
- レベル判定が実装詳細に傾く → 低

## 参照ファイル
- `references/taxonomy.md` … 種別・優先度・確信度の定義
- `references/criteria.md` … 反映すべき/すべきでないの判断基準（シグナルS1〜S10、★随時更新）
- `references/checklist.md` … 走査用の観点チェックリスト
- `references/examples.md` … 良例/反例（few-shot、★随時更新）
- `references/ingestion.md` … 混在形式の読み取り・変換ガイド
- `templates/review-table.md` … 出力テンプレ
- `feedback/README.md` … フィードバックの記録と蒸留手順
