# 判断基準：基本設計（下流）→ 要件定義書（上流）

「基本設計に書かれているが、要件定義に**無い／食い違う／曖昧なまま**」のうち、
**上流へ反映すべきもの**を見極めるための基準。

## 1. 使い方

1. 基本設計を要件カテゴリ（§3）ごとに走査する。
2. 各記述について `taxonomy.md` の**レベル判定**を行う。
3. 要件レベルなら、要件定義の対応箇所と突き合わせ、§4/§5 のシグナルに当てはめる。
4. 当てはまれば候補化し、種別・優先度・確信度・判断理由を付ける。
5. §6 のグレーゾーン・§7 の除外ルールで確信度や採否を調整する。

## 2. 大原則

- **What と How を分ける**: 要件（What/制約/前提）は上流、実現手段（How）は設計内。
- **トレーサビリティ必須**: 候補には必ず「基本設計のどこ（章・項番）」を根拠として示す。
- **断定しない**: 不確かなものは確信度=中で出し、人間の判断に委ねる。捏造・過剰提案をしない。
- **本文の機密はログに残さない**: 候補の根拠は「場所＋要旨」で表現する。

## 3. 走査する要件カテゴリ

機能要件 / 非機能要件（性能・可用性・拡張性・運用保守・移行）/ セキュリティ要件 /
データ要件 / 外部システム連携 / 業務フロー・例外/異常系 / 前提・制約 / スコープ・対象範囲 / 用語定義。

## 4. 「上流へ反映すべき」シグナル（候補化する）

| # | シグナル（基本設計側に現れるもの） | 想定種別 | 既定の確信度 |
|---|---|---|---|
| S1 | 設計で前提とした機能・条件が、要件定義に記載がない | 追加 | 中〜高 |
| S2 | 非機能の数値（性能・同時接続・保持期間・SLA等）が設計で確定したが要件は曖昧/未記載 | 明確化/追加 | 高 |
| S3 | 要件定義の記述と設計の確定内容が食い違う（数量・範囲・対象） | 変更 | 高 |
| S4 | 要件どうし、または要件と制約が設計上両立しないと判明 | 矛盾(要修正) | 高 |
| S5 | 外部システム連携・IF・データ授受が設計で必要と判明（要件未記載） | 追加 | 中〜高 |
| S6 | 業務フローの例外・異常系・エラー処理が設計で具体化（要件未記載） | 追加/明確化 | 中 |
| S7 | 要件の曖昧語（「速やかに」「適切に」「など」）を設計で具体定義した | 明確化 | 中〜高 |
| S8 | 設計の制約（性能限界・依存・前提環境）が前提条件・体制・コスト・スケジュールに波及 | 影響波及 | 中 |
| S9 | 要件にあるが設計で不要/未使用と確認された機能 | 削除・縮小 | 中 |
| S10 | 用語の意味が要件と設計でズレている（同語異義/異語同義） | 明確化 | 中 |

## 5. 重点観点（見落としやすい）

- **非機能要件**: 機能は気づきやすいが、性能・可用性・セキュリティ・運用は設計で初めて数値化されがち（S2）。
- **異常系・例外**: 正常系だけ要件にあり、設計で増えた例外処理が上流に無いことが多い（S6）。
- **暗黙の前提**: 設計が「当然」としている前提（同時利用者数、データ量、対象ブラウザ等）（S1, S8）。
- **境界・スコープ**: 設計で「対象外」と書かれた範囲が要件と不一致（S3, S9）。

## 6. グレーゾーンの扱い（確信度=中で出す）

- 要件レベルか実装詳細か判断が割れるもの → 確信度=中、判断理由に両論を書く。
- 「要件にすべきか、設計の補足で足りるか」が文脈依存のもの。
- 影響波及（S8）は事実の指摘に留め、対応はPM判断に委ねる。

## 7. 除外ルール（候補にしない／確信度=低）

実装詳細レベルとして**原則除外**:
- 画面の具体レイアウト・部品配置・色・文言の細部
- DB物理設計（テーブル・カラム・インデックス）、内部クラス/関数構成
- アルゴリズム・内部処理ロジックの選択
- 採用ライブラリ・フレームワーク・ミドルの選定（要件の非機能に波及しない限り）

例外: 上記でも**要件の非機能（性能・セキュリティ・運用）や合意事項に影響**する場合は候補化（確信度は影響の明確さに応じる）。

## 8. 候補1件あたりに必ず付ける情報

- 種別 / 対象箇所（要件定義の章・項。無ければ「記載なし」）/ 現状 / 反映案 /
  下流根拠（基本設計の章・項＋要旨）/ 優先度 / 確信度 / 影響・波及 / 判断理由。
