# 観点チェックリスト：基本設計 → 要件定義

抽出の網羅性を担保するためのスキャン用チェックリスト。
基本設計を読みながらカテゴリごとに「対応する要件記述があるか？無ければ候補化」を確認する。

> 各項目は `criteria.md` のシグナル（S1〜S10）に対応。チェックの結果、当てはまれば候補表へ。

## 機能要件
- [ ] 設計が前提にしている機能・操作で、要件定義に無いものはないか（S1）
- [ ] 要件にある機能で、設計で不要/未使用になったものはないか（S9）
- [ ] 機能の対象範囲・件数・条件が要件と設計でズレていないか（S3）

## 非機能要件（性能・可用性・拡張性・運用保守・移行）
- [ ] 性能（応答時間・スループット・同時接続・処理件数）の数値が設計で確定したが要件は曖昧/未記載でないか（S2）
- [ ] 可用性・稼働時間・障害復旧目標（RTO/RPO）が設計で前提化されているか（S2）
- [ ] データ保持期間・容量・増加見込みの前提が要件に無いか（S1, S2）
- [ ] 運用・保守・監視・バックアップの要件が設計で発生していないか（S1）
- [ ] 移行（既存データ・並行稼働）の前提が要件に無いか（S1）

## セキュリティ要件
- [ ] 認証・認可・権限の粒度が設計で具体化され、要件と差がないか（S2, S7）
- [ ] 暗号化・監査ログ・個人情報の扱いが設計で前提化され、要件に無いか（S1, S2）

## データ要件
- [ ] 必須項目・桁・形式・コード体系が設計で確定し、要件の曖昧さを埋めていないか（S7）
- [ ] マスタ・参照データの所在/管理責任が設計で前提化されていないか（S1）

## 外部システム連携
- [ ] 連携先・IF・プロトコル・授受データ・頻度が設計で必要と判明し、要件に無いか（S5）
- [ ] 連携相手側の制約・前提（稼働時間・SLA）が前提条件に波及しないか（S8）

## 業務フロー・例外/異常系
- [ ] 正常系のみ要件にあり、設計で増えた例外・エラー処理が上流に無いか（S6）
- [ ] 業務ルール・判定条件が設計で具体化され、要件の曖昧語を埋めていないか（S6, S7）

## 前提・制約・スコープ
- [ ] 設計が「当然」としている暗黙の前提（利用者数・データ量・対象環境/ブラウザ）はないか（S1, S8）
- [ ] 設計で「対象外」とした範囲が要件のスコープと矛盾しないか（S3, S9）
- [ ] 設計上の制約がコスト・スケジュール・体制・リスクに波及しないか（S8）

## 用語・整合性
- [ ] 同じ用語が要件と設計で違う意味で使われていないか（同語異義）（S10）
- [ ] 違う用語で同じ概念を指していないか（異語同義）（S10）

## 最終確認
- [ ] 各候補に「下流根拠（基本設計の章・項）」を付けたか
- [ ] 実装詳細レベル（`criteria.md` §7）を誤って要件候補にしていないか
- [ ] 確信度と判断理由が再現可能な根拠に基づくか
