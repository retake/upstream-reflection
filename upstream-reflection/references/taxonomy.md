# タクソノミー（種別・優先度・確信度の定義）

レビュー表の各列で使う分類の定義。判断のブレを減らすための共通言語。
※本ファイルは安定（あまり変えない）。実際の線引きの蓄積は `criteria.md` / `examples.md` 側で育てる。

---

## 1. レベル判定（最重要の軸）

候補を出す前に必ず「これは**要件レベル**か**実装詳細レベル**か」を判定する。

- **要件レベル**: 「何を満たすべきか（What / 制約 / 前提）」。発注者・利用者と合意すべき事柄。
  → 上流（要件定義）へ反映候補にする。
- **実装詳細レベル**: 「どう実現するか（How / 設計手段）」。要件に影響しない限り設計内に留める。
  → 原則として候補にしない（出すなら確信度=低で「念のため」）。

判定の目安: 「この記述が変わったら、発注者・利用者にとって**約束（要件）が変わる**か？」
- 変わる → 要件レベル
- 変わらない（実現手段が変わるだけ） → 実装詳細レベル

## 2. 種別（候補のタイプ）

| 種別 | 定義 | 基本設計→要件定義での例 |
|---|---|---|
| 追加 | 上流に記載がない要件・前提が下流で判明 | 設計で必要と判明した非機能要件、未記載の外部連携 |
| 変更 | 上流の記述が下流の確定内容と食い違い、上流を更新すべき | 想定処理件数が設計で見直された |
| 矛盾(要修正) | 上流の要件が下流で実現不可/論理矛盾。要件側の是正が必要 | 要件Aと要件Bが両立不能と設計で判明 |
| 明確化 | 上流の曖昧記述を下流で確定。曖昧さ解消のため上流を具体化 | 「速やかに」を「3秒以内」と設計で確定 |
| 削除・縮小 | 上流の要件が不要/過剰と判明。スコープ縮小 | 使われない要件が設計で落ちた |
| 影響波及 | 下流の確定がコスト・スケジュール・リスク・他要件に波及 | 設計で判明した制約が前提条件・体制に影響 |

## 3. 優先度（高 / 中 / 低）

「影響度 × 緊急度」で判断。

| 優先度 | 目安 |
|---|---|
| 高 | 放置すると後戻り・手戻りコスト大、または合意・契約・安全に関わる（矛盾、スコープ変更、主要非機能要件） |
| 中 | 反映すべきだが当面の致命度は低い（明確化、補足的な追加） |
| 低 | あれば望ましい程度、または実装詳細寄りで反映要否が微妙 |

## 4. 確信度（高 / 中 / 低）

「これは本当に**上流へ反映すべき**か」への自信。**精度向上ループの中心指標**。

| 確信度 | 目安 | レビュー時の扱い |
|---|---|---|
| 高 | 明確に要件レベルで、上流に欠落/矛盾が明白 | ほぼ採用想定。却下されたら criteria を疑う |
| 中 | 要件レベルかグレー、文脈依存で判断が割れうる | 人間の判断を仰ぐ主戦場。FB を最優先で学習 |
| 低 | 実装詳細寄りだが念のため提示 | 却下が基本。多発するなら抽出条件を絞る |

### 確信度の付け方（再現性のため）
- 「要件レベルである根拠」と「上流に欠落/矛盾している根拠」の**両方が明示できる** → 高
- どちらか一方が弱い／文脈依存 → 中
- レベル判定が実装詳細に傾く → 低

## 5. 採否（ユーザ記入・学習の入力）

| 値 | 意味 |
|---|---|
| 採用 | 提案どおり上流へ反映する |
| 修正採用 | 趣旨は採るが文面・範囲を直して反映 |
| 却下 | 反映しない（理由を FB に記す：実装詳細だった/既出/誤読 など） |

→ 「確信度高なのに却下」「確信度低なのに採用」は criteria 改善の重要シグナル。
