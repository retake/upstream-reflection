# 事例集（few-shot）：基本設計 → 要件定義

判断のブレを減らすための「良例（反映すべき）」「反例（反映不要＝除外）」「確信度キャリブレーション」。

## 1. 良例（上流へ反映すべき）

### 良例1：曖昧語の数値化（明確化・確信度高）
- 下流: 「主要画面の応答時間は3秒以内、同時接続100」
- 上流の現状: 「快適に操作できる応答性」
- 判断: 要件レベルの非機能を設計で数値確定。要件が曖昧 → **明確化／確信度=高**（S2,S7）
- ポイント: 「速やかに」「適切に」「快適に」等の曖昧語が下流で数値化されたら高確度の候補。

### 良例2：要件と設計の食い違い（矛盾・確信度高）
- 下流: 「24時間利用可能（保守時間除く）」 / 上流: 「平日9:00–18:00に利用できる」
- 判断: 明確な食い違い → **矛盾(要修正)／確信度=高**（S3）。利用時間は合意事項なので優先度も高。

### 良例3：要件機能のスコープ落ち（削除・縮小・確信度高）
- 下流: 「写真表示は今回のリリース対象外」 / 上流: 「収容人数と写真を表示する」
- 判断: 要件にある機能が設計で対象外 → **削除・縮小／確信度=高**（S9）。スコープ合意の更新が要る。

### 良例4：要件に痕跡のない新機能（追加・確信度中〜高）
- 下流: 「予約確定時・開始30分前にメール通知」 / 上流: 記載なし
- 判断: 通知は利用者向けの機能であり要件級。要件に一切無い → **追加／確信度=中〜高**（S1）。
- ポイント: 「要件に痕跡ゼロ」かつ「利用者・発注者に見える機能」は見落とされやすく、価値が高い。

## 2. 反例（反映不要＝設計内に留める／除外）

### 反例1：技術選定
- 下流: 「Spring Boot / PostgreSQL を採用」
- 判断: 実現手段の選定。非機能（性能・セキュリティ等）に波及しない限り **除外**（criteria §7）。
- 例外: その選定が性能・運用・コストの要件に影響するなら、影響側だけを候補化。

### 反例2：DB物理設計
- 下流: 「reservations テーブル構成・複合インデックス」
- 判断: How の典型。要件の約束は変わらない → **除外**（criteria §7）。

### 反例3：画面レイアウトの細部
- 下流: 「検索ボックスを上部、会議室一覧を右に配置」
- 判断: 実装詳細 → **除外**。ただし「ある情報項目が画面に必要」というレベルなら機能要件候補になりうる。

## 3. 確信度キャリブレーション（グレーを中に置く）

### 中の例1：認証「方式」の要件化粒度
- 下流: 「社内SSO（AD連携）で認証」 / 上流: 「社員のみ利用可能」
- なぜ中: 「認証が要る」は要件級だが「SSOを採る」は設計判断の側面もある。
  → 確信度=中。判断理由に「要件＝認証の必要性／設計＝実現手段」と両論を書き、人間に粒度を委ねる。

### 中の例2：例外系の挙動をどこまで要件に書くか
- 下流: 「重複予約は排他制御、競合時エラー」 / 上流: 「二重予約を防止する（挙動未定義）」
- なぜ中: 防止の意図は要件にあるが、エラー挙動の粒度は設計寄り。
  → 確信度=中。「業務ルールとして要件化すべき部分」と「設計に委ねる部分」を分けて提案。
