# フィードバックと精度向上ループ

このスキルは、日々のレビュー結果を**フィードバック**として溜め、定期的に**蒸留**して
判断基準（`references/criteria.md`）と事例集（`references/examples.md`）を更新することで
精度を上げていく。配布物にベースライン知識を同梱し版管理する**ハイブリッド型**。

```
  レビュー実行 ──▶ レビュー表（確信度＋判断理由つき）
                          │
        ユーザが「採否＋FB理由」を記入
                          │
              ローカルのフィードバックログに蓄積
                          │   （各PCで個別に溜まる）
        ┌─────────────────┘
   定期的にメンテナが「蒸留」
        │
   criteria.md / examples.md の更新案 ──▶ レビューして反映 ──▶ CHANGELOG・版上げ ──▶ 再配布
```

## 1. フィードバックの記録（利用者）
1. レビュー表の各行に **採否（採用／修正採用／却下）** と **FB理由** を記入する。
2. `feedback-log.template.md` をコピーしてローカルに追記する（例: `feedback/log_2026-Q2.md`）。
   - **配布物の外**、または個人の作業領域に保管する。リポジトリに混ぜない。
3. **機密保持**: ログに資料の本文を貼らない。根拠は「資料名＋章・項＋要旨」で表す。

## 2. 学習に効くシグナル（記録時に意識）
- **過検出**: 確信度=高なのに却下された → criteria のシグナルが強すぎる／除外漏れ。
- **見落とし（漏れ）**: 表に出ず、後工程で「要件に足すべきだった」と判明 → シグナル追加が必要。
- **誤った確信度**: 確信度=低が採用、または=高が頻繁に却下 → キャリブレーションを直す。
- **繰り返しパターン**: 同種の却下／採用が何度も起きる → 事例集に代表例として固定化。

## 3. 蒸留の手順（メンテナ。月次 or 10レビューごと等）
Claude に「このフィードバックログを蒸留して criteria/examples の更新案を出して」と依頼し、以下を実施:

1. **集計**: ログを種別・確信度・採否で集計し、§2 のシグナルを洗い出す。
2. **更新案の作成**:
   - `criteria.md`: シグナル S1〜S10 の表現・既定確信度の調整、除外ルール（§7）の追記。
   - `examples.md`: 代表的な良例／反例／キャリブレーション例の追加・差し替え（一般形で）。
3. **レビュー**: 更新案を人間が確認（過学習・特定案件依存になっていないか）。
4. **記録**: `criteria.md` と `examples.md` の改訂ログに1行、`CHANGELOG.md` にバージョンを追記。
5. **再配布**: `upstream-reflection/` フォルダを配布し、各PCを新版に置き換える。

## 4. バージョニング方針
- 知識（criteria/examples）の更新ごとに `CHANGELOG.md` でマイナー版を上げる。
- レビュー表ヘッダの「スキル版」に版と criteria 改訂日を記録 → どの知識で出た候補か追跡できる。

## 5. 受け入れ確認（任意だが推奨）
- 知識を更新したら、`samples/` の合成ペアで再実行し、再現率（E1〜E9）と除外（X1〜X3）が
  劣化していないかを確認してから配布する（リグレッション防止）。
