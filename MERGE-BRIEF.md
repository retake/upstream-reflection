# マージ指示書（このファイルを読んだ Claude Code は、ここから作業を開始してください）

> **あなた（このリポジトリをクローンした環境＝業務PC側の Claude Code）への指示書です。**
> ユーザがこのファイルを開いた／あなたに読ませた時点で、下記の手順に従って
> **2つのスキルの「比較」→「マージ（統合版の作成）」** を開始してください。
> 破壊的操作（既存スキルの上書き・削除、勝手な push）は確認を取ってから行うこと。

## 依頼の要約
「下流成果物を上流資料に突き合わせ、上流への反映候補をレビュー表で抽出する」という同種のスキルが
2つあります。両者を比較し、良いとこ取りで **1つの統合版** にまとめてください。

## 対象スキル
- **A（ローカル開発版）**: この業務PCで開発中のスキル。場所が不明なら、まず
  `~/.claude/skills/` 配下や作業フォルダを探し、見つからなければ**ユーザに場所を尋ねる**。
- **B（参照実装）**: このリポジトリ（あなたが今クローンしたもの）。**配布本体は `upstream-reflection/`**
  フォルダ（`SKILL.md`, `references/`, `templates/`, `feedback/`）。
  `samples/`＝合成検証データ、`PROJECT_PLAN.md`＝計画書、本ファイル `MERGE-BRIEF.md`＝この指示書。
  - まず B の実ファイル（特に `upstream-reflection/SKILL.md` と `upstream-reflection/references/criteria.md`,
    `taxonomy.md`, `checklist.md`, `examples.md`, `ingestion.md`, `templates/review-table.md`,
    `feedback/README.md`）を読むこと。

## B の設計思想（読む前の地図。詳細は実ファイル参照）
- **目的/出力**: 下流→上流の反映候補を「レビュー表」で提示。列＝対象箇所/現状/反映案/根拠(下流トレーサビリティ)/
  優先度/確信度/判断理由。対応ペア例＝基本設計→要件定義。
- **判断の核**: 「設計に在るが要件に無い／矛盾／曖昧」を拾い、**What(要件)とHow(実装詳細)で仕分け**。
  実装詳細は除外。判断基準は `references/criteria.md` のシグナル **S1〜S10** と除外ルール(§7)。
- **精度向上（ハイブリッド型学習）**: 全候補に**確信度＋判断理由**を必須付与 → フィードバック(採否＋理由)を
  蓄積 → 定期**蒸留**で `criteria.md`/`examples.md` を更新 → `CHANGELOG.md` で版管理し再配布。
- **携帯性**: 自己完結・個人情報なし・混在形式はグレースフル変換(`ingestion.md`)。
- **検証**: `samples/` の合成ペアで再現率・除外トラップを確認できる。

## 手順（このまま実行）
1. **A の特定**: A の場所と構成を確認（不明ならユーザに確認）。A・B 双方の SKILL.md と
   付随ファイル（判断基準・テンプレ・事例・学習の仕組み）を読む。
2. **比較・採点**: 次の観点で A/B を採点し、**良い／悪いを表に整理**:
   - トリガ精度（description が用途に正しく反応するか）
   - ワークフローの再現性（手順に抜け・曖昧さがないか）
   - 判断基準の質（What/How 分離、反映要否の線引きの明確さ）
   - 出力形式（現場で使える形か、トレーサビリティがあるか）
   - 精度向上ループ（フィードバック→学習の仕組みがあるか）
   - 配布・携帯性（自己完結・環境非依存・機密配慮）
   - 事例(few-shot)・検証（良例/反例、受け入れ確認の有無）
   - 保守性・拡張性（版管理、他レベルペアへの拡張のしやすさ）
3. **マージ**: 比較を踏まえ、両者の強みを取り込んだ**統合版スキル**を作る。
   - 良い構造の方をベースにし、もう片方の優れた観点・基準・事例・手順を吸収する。
   - 重複は統合、矛盾は明示して解消。**どちらの強みも失わないこと**。
   - 統合版は自己完結フォルダにし、`CHANGELOG.md` に統合の経緯を記録。
4. **受け入れ確認**: 可能なら `samples/`（または A 側の検証データ）で統合版を走らせ、
   再現率・除外トラップが劣化していないか確認する。
5. **サマリ**: 「**A から採った点／B から採った点／新規追加／捨てた点と理由**」を明記。

## 出力
- A/B の比較表（それぞれの良い・悪い）
- 統合版スキル一式（**新ブランチ推奨**。既存スキルは上書きせず別ブランチ／別フォルダに）
- 統合サマリ（上記5の内容）

## 注意
- 既存スキル（A）を上書き・削除しない。統合版は別ブランチ／別フォルダに作り、反映可否はユーザが決める。
- 資料・スキルの内容を外部サービスへ送信しない。機密はログ等に残さない。
- push は明示の指示があるまで行わない。
